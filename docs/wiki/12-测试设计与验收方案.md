# 12. 测试设计与验收方案

> 文档类型：项目质量文档  
> 同步时间：2026-03-13  
> 目标：说明系统如何验证正确性、稳定性和主流程可靠性

## 1. 测试策略

本项目采用“单元测试 + 编译检查 + 业务冒烟清单”的组合方式：

1. 前端通过静态检查验证类型、语法与未使用变量问题。
2. 后端通过编译检查验证接口、DTO、控制器和服务层改动是否一致。
3. 关键服务使用 Mockito 单元测试，覆盖核心业务规则。
4. 业务链路通过人工冒烟脚本验证前后台闭环是否可稳定复现。

## 2. 测试范围

### 2.1 前端

| 测试项 | 目标 |
|---|---|
| 类型与语法检查 | 保证页面改动不会破坏构建 |
| 交互反馈检查 | 关键流程不再依赖原生 `alert` |
| 页面状态检查 | loading / empty / error 均有可视反馈 |

### 2.2 后端

| 测试项 | 目标 |
|---|---|
| AuthService | 验证注册、重复用户名、登录成功、密码错误 |
| ProductService | 验证分类输出与商品分页排序映射 |
| CartService | 验证库存校验、首次加购、同商品数量合并 |
| OrderService | 验证创建订单、库存不足、支付成功、自动流转 |

## 3. 单元测试设计

### 3.1 认证服务

| 用例 | 断言 |
|---|---|
| 注册成功 | 返回 token，用户信息正确 |
| 重复用户名 | 抛出业务异常，错误码为 `4002` |
| 登录成功 | 返回 token，用户角色和昵称正确 |
| 密码错误 | 抛出业务异常，错误码为 `4003` |

### 3.2 商品服务

| 用例 | 断言 |
|---|---|
| 分类查询 | 仅返回启用分类的响应模型 |
| 商品分页排序 | `Pageable` 中页码、条数、排序字段正确 |

### 3.3 购物车服务

| 用例 | 断言 |
|---|---|
| 新增购物车条目 | 保存数量正确，用户绑定正确 |
| 库存不足 | 抛出 `4005` 业务异常 |
| 同商品合并数量 | 已有条目数量正确累加 |

### 3.4 订单服务

| 用例 | 断言 |
|---|---|
| 创建订单成功 | 总金额正确、库存减少、销量增加、购物车移除 |
| 库存不足拒绝下单 | 不创建订单，抛出 `4005` |
| 模拟支付 | 状态变更为 `PAID`，支付时间回写 |
| 自动流转 | `PAID -> SHIPPED`，`SHIPPED -> COMPLETED` |

## 4. 业务冒烟验收清单

### 4.1 普通用户链路

1. 使用 `demo / 123456` 登录。
2. 在首页或搜索页选择商品。
3. 进入商品详情页，加入购物车。
4. 在购物车页创建订单。
5. 在支付页完成模拟支付。
6. 在订单页查看支付时间、发货时间和完成状态。

### 4.2 管理员链路

1. 使用 `admin / admin123` 登录。
2. 进入后台仪表盘查看概览数据。
3. 打开订单管理页查看详情抽屉。
4. 执行发货或完成操作。
5. 返回前台订单页确认状态变化。

## 5. 验收命令

```bash
# 前端
npm run lint

# 后端编译
cd server
mvn -q -DskipTests compile

# 后端单元测试
cd server
mvn -q test
```

## 7. 可直接复用的结论

可以在项目总结中表述为：

- 测试设计围绕核心交易链路展开，兼顾自动化验证与人工验收。
- 由于项目当前以核心业务闭环为重点，测试重点放在核心业务正确性和主流程稳定性，而非大规模压力测试。
