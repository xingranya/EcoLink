# 16. 常见问题说明

> 文档类型：项目说明文档  
> 使用方式：可按“结论 + 原因 + 当前实现 + 可扩展方向”四段式组织说明

## 1. 为什么选择 Vue 3 + Spring Boot？

推荐回答：

“因为本项目是典型的前后端分离场景。Vue 3 适合快速构建用户端和后台管理端页面，组件化和状态管理比较清晰；Spring Boot 在接口开发、权限控制、事务处理和数据库集成上比较成熟。两者组合适合当前项目规模，也便于保持工程结构清晰。”

## 2. 为什么使用 JWT？

推荐回答：

“JWT 适合前后端分离场景，前端只需要在请求头携带令牌，后端就可以识别当前用户身份。这样前端和后端之间不会依赖服务端 Session，更方便扩展多端访问。项目里用户接口和管理员接口都基于 JWT 做身份识别和角色判断。”

## 3. 为什么用 Flyway？

推荐回答：

“Flyway 的价值在于把数据库结构和初始化数据版本化。这样我在重建数据库时，不需要手动执行多份 SQL，只要运行迁移脚本就能得到一致的表结构、默认账号和初始化数据。它对项目环境复现和团队协作都很有帮助。”

## 4. 如何保证库存一致性？

推荐回答：

“当前项目在创建订单时会重新校验购物车商品库存，并在同一事务里完成订单创建、订单明细生成、库存扣减和销量增加。这样可以避免用户绕过购物车页面直接使用旧库存下单。对于当前项目规模，这种方案是足够的；如果将来扩展到更高并发，可以继续加乐观锁或库存预扣机制。”

## 5. 为什么要做订单快照？

推荐回答：

“订单项中保存了 `productName`、`productImage` 和 `salePrice` 等快照字段，而不是完全依赖商品主表实时读取。这样即使后续商品名称、价格或图片发生变化，历史订单也能保持下单时的真实信息，便于订单追溯和售后说明。”

## 6. 前后端是如何解耦的？

推荐回答：

“前端只依赖接口契约和类型定义，不直接依赖数据库结构；后端通过 DTO 和统一响应模型向前端输出数据。比如订单字段、后台统计字段、商品详情字段都通过接口统一定义，前端再在 `types/api.ts` 中进行映射，这样两端职责比较清晰。”

## 7. 后台权限如何控制？

推荐回答：

“后台权限分两层：第一层是前端路由守卫，只有登录且角色为管理员的用户才能进入 `/admin`；第二层是后端 Spring Security，对 `/api/v1/admin/**` 路径做管理员访问控制。这样即使前端被绕过，后端接口也不会对普通用户开放。”

## 8. 为什么不接真实支付？

推荐回答：

“本项目重点是完整业务闭环、架构设计和订单状态流转。真实支付涉及商户申请、回调验签、对账和安全审计，超出了当前项目范围。因此我采用模拟支付来保留订单状态流转和支付后续逻辑，使系统边界更清晰，也便于后续扩展真实支付能力。”
