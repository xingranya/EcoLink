# 11. 非功能需求与质量目标

> 文档类型：项目分析文档  
> 同步时间：2026-03-13  
> 目标：说明系统除了“能用”之外，还要在安全性、稳定性、可维护性和可用性上达到什么标准

## 1. 非功能需求概览

| 维度 | 目标 | 当前落地方式 |
|---|---|---|
| 安全性 | 用户身份可识别、管理员能力不可越权 | JWT + Spring Security + 路由守卫 |
| 稳定性 | 主链路不能因空白页、原生弹窗或字段缺失中断使用 | loading / empty / error 展示、统一消息提示 |
| 可维护性 | 前后端边界清晰、字段命名统一、文档可追踪 | TypeScript 类型、DTO、Wiki 字段字典 |
| 可复现性 | 可快速恢复默认数据与账号 | Flyway 初始化脚本 |
| 易用性 | 用户链路反馈及时、后台操作可见 | Toast 提示、订单进度条、后台详情抽屉 |

## 2. 安全性要求

### 2.1 认证与授权

- 所有登录态接口通过 JWT 令牌识别用户身份。
- 普通用户与管理员基于角色区分权限范围。
- 前端路由层通过 `requiresAuth` 与 `requiresAdmin` 控制页面访问。
- 后端接口层通过 Spring Security 拦截后台管理路径。

### 2.2 数据访问边界

- 地址、购物车、订单等用户资源仅允许当前登录用户访问。
- 后台管理接口仅允许管理员操作，不与普通用户接口混用。
- 订单详情查询同时校验 `id` 和 `userId`，避免越权读取。

## 3. 稳定性要求

### 3.1 页面状态稳定

- 关键页面必须具备加载态，避免请求过程中出现白屏。
- 数据为空时应具备空态说明，避免用户误认为系统失效。
- 请求失败时应有统一错误提示，而不是浏览器原生 `alert`。

### 3.2 链路稳定

- 未登录访问受保护路由时，系统应自动跳转登录页，并保留回跳地址。
- 支付成功后应自动跳转订单页，减少用户手工切换。
- 管理员登录后顶部导航和移动端菜单均有后台入口。

## 4. 可维护性要求

### 4.1 前端

- 使用 TypeScript 定义接口响应类型，减少字段漂移。
- 将消息反馈收敛到统一 `toast` store，避免页面各自维护提示逻辑。
- 使用 Vue Router 与 Pinia 进行职责分离。

### 4.2 后端

- 控制器负责参数接收和响应封装，业务逻辑下沉到 Service。
- Repository 层使用 JPA 接口进行数据访问，降低样板代码。
- 迁移脚本统一管理数据库结构和种子数据。

## 5. 可用性与可观察性要求

项目在功能实现之外，还需要保证主流程稳定、反馈清晰，因此专门提出以下约束：

1. 默认账号必须清晰可见，便于快速切换普通用户与管理员角色。
2. 主流程必须有一致的成功反馈，例如登录成功、加购成功、支付成功、发货成功。
3. 首页、商品详情页、后台仪表盘应同时承担“功能入口”和“信息总览”的作用。

## 6. 质量目标分解

| 质量目标 | 验收标准 |
|---|---|
| 前端类型稳定 | `npm run lint` 通过 |
| 后端编译稳定 | `mvn -q -DskipTests compile` 通过 |
| 关键后端流程可验证 | 认证、商品筛选、购物车、订单服务具备单元测试 |
| 文档一致性 | README、Wiki、字段字典与相关说明文档彼此可跳转 |

## 7. 项目总结建议

项目说明中可以将本章写成“系统质量属性设计”，强调：

- 本项目不只关注功能实现，还特别考虑了主流程稳定性和环境复现性。
- 质量目标与实现策略一一对应，具有可验证依据。
